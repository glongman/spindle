Option to veto an incremental build

Identifier:
com.iw.plugins.spindle.core.incrementalBuildVeto

Since:
Spindle 3.1.9

Description:

Developers can register a veto on incremental builds. The canonical example of when this might be needed is when special runtime classes are being used so that scripts in other launguages are being used to act as Page or Component classes.

Basicly the veto works like this. A developer registers a file extension string. When an incemental build is started, the resource change set will be checked for files with this extension that have changed. If this is the case, the incremental build will not occur. Rather a full build will be initiated instead.

Change in this case means a create, delete, move or a rename (to or from). Change delta's that indicate that the only the file's content has changed are ignored.

Spindle will ignore contributions that try to register veto for the following extensions:

  • application
  • library
  • jwc
  • page
  • html

As well, even if you register the extension "xml", web.xml is not considered.

This point is to be considered temporary until full index/refactoring is implemented. At that point this extension will no longer be needed and any contributions will be ignored.

Last point, every contribution to this extension will slow down the build. Use sparingly.

Configuration Markup:

<!ELEMENT extension (veto+)>

<!ATTLIST extension

point CDATA #REQUIRED

id    CDATA #IMPLIED

name  CDATA #IMPLIED>


<!ELEMENT veto EMPTY>

<!ATTLIST veto

extension CDATA #REQUIRED

id        CDATA #REQUIRED

name      CDATA #REQUIRED>

  • extension - When a file in the project that has this extension changes, an incremental build is vetoed and a Full Build is required. Don't include the 'dot' character.
  • id -
  • name - Give a user friendly name. Eventually, users will be able to enable/disable vetoes.
